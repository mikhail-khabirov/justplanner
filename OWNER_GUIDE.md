# 📋 Инструкция администратора JustPlanner

Привет! Здесь всё что нужно знать чтобы управлять сервисом — менять настройки, ключи, пароли. Технических знаний почти не требуется, но придётся пользоваться терминалом. Буду пояснять каждый шаг.

## 🌐 Что у тебя есть

- **Сайт:** https://justplanner.ru
- **Сервер:** `72.56.35.143` (Ubuntu Linux, в Москве)
- **База данных:** PostgreSQL — хранит всех пользователей, задачи, подписки
- **Один файл конфигурации** на сервере — туда вносятся все ключи и настройки

## 🔌 Как подключиться к серверу

### На Mac
1. Открой **Terminal** (Cmd+Space → "Terminal" → Enter)
2. Введи команду и нажми Enter:
   ```
   ssh root@72.56.35.143
   ```
3. Введи пароль root (уточнить у предыдущего владельца). При вводе пароля курсор не двигается — это нормально, печатай вслепую и жми Enter.
4. Если успешно — увидишь строку `root@msk-1-vm-b066:~#`. Ты внутри сервера.

### На Windows
1. Скачай **PuTTY**: https://www.putty.org/
2. Открой PuTTY, в поле **Host Name** введи `72.56.35.143`, нажми **Open**
3. Логин: `root`, пароль: (тот же)

### Чтобы выйти
Введи `exit` и нажми Enter.

## 📝 Где находится конфигурация

Один файл: `/var/www/justplanner/server/.env`

Там сгруппированы все секреты и настройки: API-ключи Google, YooKassa, SMTP, Telegram-бот, пароли админ-панели и т.д. Каждая секция помечена комментарием на русском.

### Открыть файл
После того как зашёл по SSH:
```
nano /var/www/justplanner/server/.env
```

Откроется текстовый редактор. Стрелками двигайся по строкам, набирай текст как в обычном Word.

### Сохранить и выйти
- **Ctrl+O** → нажми Enter (сохранить)
- **Ctrl+X** (выйти из редактора)

### Применить изменения
После сохранения **обязательно** перезапусти бэкенд:
```
pm2 restart justplanner-api
```

Без этой команды изменения не подхватятся.

### Проверить что не сломалось
```
pm2 logs justplanner-api --lines 20
```
Смотри последние строки. Если видишь `🚀 Server running` и нет красных `error` — всё хорошо. Чтобы выйти из режима логов — Ctrl+C.

## 🔧 Сценарии

### Сценарий 1: Сменить YooKassa (когда оформишь свой магазин)

1. Зайди в свой кабинет YooKassa: https://yookassa.ru/my
2. **Настройки → Магазин** → скопируй **shopId** и **Секретный ключ** (тот что начинается на `live_`)
3. Зайди на сервер, открой конфиг:
   ```
   nano /var/www/justplanner/server/.env
   ```
4. Найди строки `YOOKASSA_SHOP_ID=` и `YOOKASSA_SECRET_KEY=`, замени значения на свои.
5. Сохрани (Ctrl+O, Enter, Ctrl+X)
6. Перезапусти: `pm2 restart justplanner-api`

**ВАЖНО:** в кабинете YooKassa также нужно добавить webhook:
- **Настройки → Уведомления → HTTP-уведомления → Добавить URL**
- URL: `https://justplanner.ru/api/billing/webhook`
- События: `payment.succeeded`, `payment.canceled`, `refund.succeeded`

Без webhook оплаты будут проходить, но статус подписки в системе не обновится.

### Сценарий 1.1: ⚠️ ВАЖНО — после смены YooKassa перепривязать карты Pro-юзеров

**Зачем это нужно**

Сохранённые карты пользователей привязаны к конкретному shop_id предыдущего магазина. Когда ты сменил магазин — все эти "сохранения" перестали работать. Если ничего не делать — ровно в день следующего автопродления у каждого Pro-пользователя списание не пройдёт, и через 3 неудачные попытки он будет переведён на бесплатный тариф. Это плохой сценарий — много потерянных подписок.

**Что нужно сделать (один раз, после смены SHOP_ID/SECRET_KEY)**

В системе есть готовый скрипт, который:
1. Помечает все старые сохранённые карты как недействительные (выключает автопродление)
2. **Pro-доступ у пользователей сохраняется** до конца текущего оплаченного периода
3. Каждому Pro-пользователю отправляется красивое email-письмо с просьбой заново привязать карту

После этого пользователь:
- Открывает сайт → Настройки → Подписка → "Привязать карту"
- Вводит данные карты, списывается и сразу возвращается 1 ₽
- Автопродление снова включено, теперь уже через **твой** YooKassa

**Как запустить кампанию**

⚠️ Сначала убедись, что:
- ✅ Новые `YOOKASSA_SHOP_ID` и `YOOKASSA_SECRET_KEY` уже прописаны в `.env`
- ✅ В кабинете нового YooKassa настроен webhook на `https://justplanner.ru/api/billing/webhook`
- ✅ Сделан `pm2 restart justplanner-api`

Иначе пользователи нажмут "Привязать карту", и привязка не пройдёт.

Подключись к серверу и запусти:

```
ssh root@72.56.35.143
cd /var/www/justplanner/server
node scripts/yookassa-rebind-campaign.js --dry-run
```

Покажет сколько Pro-юзеров будет затронуто и первые 5 email — это **черновой запуск**, ничего не меняется.

Если число выглядит правильно — запусти по-настоящему:

```
node scripts/yookassa-rebind-campaign.js
```

Скрипт:
- Очищает старые payment_method_id (без потери Pro-доступа)
- Шлёт email каждому пользователю по очереди (с паузой ~0.5 сек между письмами)
- В конце выводит статистику: сколько отправлено, сколько с ошибкой

Запуск длится примерно 1 минуту на 100 пользователей.

**Что увидит пользователь**

Письмо приходит с темой: *"Важно: подтверди карту, чтобы Pro продолжил работать"*. Внутри — объяснение почему нужно заново привязать карту, инструкция в 3 шага и большая кнопка "Перейти и привязать карту".

**Если пользователь не перепривяжет карту**

Pro работает до конца текущего периода (current_period_end). После этого автоматически переключается на бесплатный план. Можно повторно запустить email через несколько дней, но скрипт уже очистил `yookassa_subscription_id`, поэтому повторный запуск не найдёт подходящих юзеров. Лучше тогда послать напоминание через любой свой почтовый сервис вручную.

**Сколько обычно возвращается**

По опыту индустрии — около 50-60% активных пользователей перепривязывают карту в первую неделю после рассылки. Кто давно не пользуется — скорее всего не вернётся.

### Сценарий 2: Сменить email-провайдер (SMTP)

Email-сервис нужен для: писем с кодом подтверждения, welcome-писем, сброса пароля.

1. Возьми у нового SMTP-провайдера (Yandex 360, Beget, SendGrid и т.д.) данные:
   - адрес сервера (например `smtp.yandex.ru`)
   - порт (обычно `465` или `587`)
   - email и пароль для отправки
2. Открой конфиг: `nano /var/www/justplanner/server/.env`
3. Замени значения у строк:
   ```
   SMTP_HOST=...
   SMTP_PORT=...
   SMTP_USER=...
   SMTP_PASS=...
   ```
4. Сохрани, перезапусти: `pm2 restart justplanner-api`
5. Проверь — попроси кого-нибудь зарегистрироваться на сайте. Если код подтверждения пришёл — работает.

### Сценарий 3: Сменить Google OAuth

Если решишь не использовать Google-проект, переданный предыдущим владельцем, а сделать свой:

1. Зайди в https://console.cloud.google.com
2. Создай новый проект, в нём:
   - **APIs & Services → OAuth consent screen** → External, заполни поля, **Publish App**
   - **APIs & Services → Credentials → Create Credentials → OAuth Client ID**
   - Тип: **Web application**
   - В **Authorized redirect URIs** добавь: `https://justplanner.ru/api/auth/google/callback`
3. Скопируй **Client ID** и **Client Secret**
4. На сервере в конфиге замени:
   ```
   GOOGLE_CLIENT_ID=...
   GOOGLE_CLIENT_SECRET=...
   ```
5. Сохрани, перезапусти `pm2 restart justplanner-api`

**Важно:** при следующем входе все пользователи, входящие через Google, увидят окно "Разрешить приложению доступ" один раз — это нормально. Их аккаунты найдутся по email и не потеряются.

### Сценарий 5: Сменить Telegram-бота для напоминаний

1. В Telegram открой [@BotFather](https://t.me/botfather), команда `/newbot`, создай нового бота
2. Скопируй токен (длинная строка с двоеточием)
3. На сервере в конфиге замени:
   ```
   TELEGRAM_USER_BOT_TOKEN=...
   TELEGRAM_USER_BOT_USERNAME=имя_твоего_бота_без_@
   ```
4. Перезапусти: `pm2 restart justplanner-api`

**Важно:** все пользователи которые уже привязали старого бота — потеряют связь. Им придётся заново привязать через Settings в приложении.

### Сценарий 7: Бэкап базы данных

Раз в неделю желательно сохранять копию базы. Команда:
```
su - postgres -c "pg_dump -Fc -d justplanner" > /root/backup-$(date +%F).dump
```
Файл сохранится в `/root/backup-YYYY-MM-DD.dump`. Скачай его себе на компьютер любым SFTP-клиентом (FileZilla, Cyberduck) или такой командой **с локального компа** (не из SSH):
```
scp root@72.56.35.143:/root/backup-*.dump ~/Downloads/
```

### Сценарий 8: Восстановить базу из бэкапа

⚠️ Опасная операция, делай только если уверен — стирает все текущие данные.
```
ssh root@72.56.35.143
su - postgres -c "dropdb justplanner && createdb -O justplanner_user justplanner && pg_restore --no-owner --role=justplanner_user -d justplanner /root/backup-2026-04-18.dump"
pm2 restart justplanner-api
```
Замени дату в имени файла на актуальную.

## 🌐 Как сменить A-запись DNS (когда переносишь домен)

Если домен `justplanner.ru` зарегистрирован у тебя — тебе нужно убедиться что A-запись указывает на сервер `72.56.35.143`.

1. Зайди в кабинет регистратора домена (например, reg.ru, beget.com, nic.ru)
2. Найди раздел **Управление DNS** или **DNS-записи** или **Настройки зоны**
3. Найди запись типа **A** для `justplanner.ru` и для `www`
4. В обеих записях должно быть значение `72.56.35.143`
5. Если другое — поменяй. Сохрани.
6. Изменения вступают в силу через 5 минут — несколько часов (зависит от провайдера и кешей).

Если в кабинете не понимаешь что нажимать — напиши в техподдержку регистратора, они помогут (это для них рутина).

## 🆘 Что делать если что-то сломалось

### Сайт не открывается
```
ssh root@72.56.35.143
pm2 list
```
Смотри статус `justplanner-api`. Если `errored` или `stopped` — попробуй:
```
pm2 restart justplanner-api
pm2 logs justplanner-api --lines 30
```
В логах будут красные строки с описанием ошибки. Скорее всего, недавно менял `.env` и где-то опечатка. Открой `.env`, проверь что не удалил знак `=` или не вставил лишнего пробела.

### Откатить ошибку в `.env`
Если совсем сломал — есть резервная копия:
```
ls /var/www/justplanner/server/.env.bak 2>/dev/null
```
Если есть `.env.bak` — восстанови:
```
cp /var/www/justplanner/server/.env.bak /var/www/justplanner/server/.env
pm2 restart justplanner-api
```

### Сертификат SSL истёк
SSL обновляется автоматически, но если вдруг не сработало:
```
certbot renew --force-renewal
systemctl reload nginx
```
